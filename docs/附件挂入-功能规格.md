# 附件（挂入文档）— 功能规格

> 在纪事中挂入本机任意格式的文件（csv / txt / pdf / xlsx…）。本项目**不解析、不打开**附件内容，
> 只负责：加密保存、在纪事里显示「有这么个文档」、交给系统默认程序打开、随导出 ZIP 一起携带。
> 目的：相关文档与纪事自成一体、方便携带，同时**不把本项目做成文档编辑器**。

配套阅读：[功能改进指引](功能改进指引.md)（架构铁律与避坑点）。

---

## 一、用户可见行为

| 位置 | 行为 |
| --- | --- |
| 编辑纪事窗口底部 | 「📎 附件」按钮 → 文件选择框 → 挂入当前纪事，成为一个内容块 |
| 附件块 | 显示 图标 + 原始文件名 + 大小；可上移/下移/删除，与图片、音频块一致 |
| 「打开」按钮 | 解密到临时目录（**保留原文件名**）→ 系统默认程序打开 |
| 「另存为」按钮 | 解密并另存到用户选定的本机路径 |
| 首页列表 | 有附件的纪事显示 `📎N`（与 `🖼N` / `🎵N` 并列） |

限制：单个附件 **50MB**（`LocalStore.MaxAttachmentBytes`）。加密是全内存操作，且附件会整份进导出 ZIP。
从 ZIP 导入既有附件时**不卡这个上限**——宁可收下也不丢数据。

## 二、存储

附件与图片/音频**走同一条路**：DPAPI（CurrentUser）加密落盘到 `media/`，文件名前缀 `file_`。

- 块模型：`BlockType.File`，复用 `NoteBlock.Url`（加密文件路径）与 `NoteBlock.Alt`（**原始文件名**）。
- 新增本地字段 `NoteBlock.Size`（字节数，仅供显示）——只进 `notes.json`，不进导出 ZIP。
- 打开附件：`MediaStore.DecryptToTempNamed` 解密到 `temp/open_<guid>/原文件名`，独立子目录避免同名互相覆盖；
  下次启动 `CleanupTemp()` 统一清理这些明文残留。

## 三、导出 / 导入（跨平台契约）

**没有新增任何 JSON 字段**，只新增了一个 `type` 取值（见铁律 1 的扩展方式）：

```jsonc
{
  "type": "file",
  "order": 2,
  "text": "[附件] 报表.csv",        // 兜底：老客户端不认识 file → 当文本块显示
  "mediaFileName": "file_9f3a.csv", // ZIP 内 media/ 下的文件名
  "alt": "报表.csv"                 // 原始文件名
}
```

ZIP 内附件与图片/音频同放 `media/`，内部明文、**整包 WinZip AES-256 密码保护**——加密强度与其他媒体完全一致。

- 附件文件丢失（源文件被删/ZIP 里缺）→ 退化成文本块 `[附件丢失] 原名`，不留空壳。
- **Android 端未适配前**：导入含附件的 ZIP 时，附件块会被当成文本块显示为 `[附件] 报表.csv`，
  文件字节仍在 ZIP 的 `media/` 里但不会被引用；若从 Android 端再导出，该附件会丢失。
  要双向可用，Android 端需补 `file` 的 type 映射。
